# خطة إعادة الهيكلة والإصلاح المحاسبي لمشروع فجر الوادي

## اسم المهمة

إعادة هيكلة منطق القاصة، الكاش، الممول، الشركة، الزبائن، الأقساط، وقيود دفتر الأستاذ في مشروع **فجر الوادي لتجارة السيارات**.

---

## وصف المشروع

أنت تعمل على مشروع قائم ومكتمل لإدارة معرض سيارات وحسابات.

التقنيات المستخدمة:

- Backend: Rust + Tauri
- Frontend: React + TypeScript
- Database: SQLite
- النظام يحتوي على سيارات، شراء، بيع، زبائن، شركاء، ممولين، شركات، مستثمرين، قاصة، أرباح، أقساط، ومصاريف.

---

## الهدف العام

إصلاح المشاكل المحاسبية في المشروع بدون تغيير غير مطلوب في شكل الواجهة أو طريقة العمل العامة.

المطلوب هو:

1. حذف مفهوم **خارج القاصة** من المشروع نهائياً.
2. جعل كل حركة نقدية حقيقية تتم فقط من خلال **قاصه**.
3. تثبيت قاعدة أن أي **إيداع** لحساب نوعه **شركة** أو **ممول** لا يدخل القاصة نهائياً.
4. تغيير مفهوم **رأس المال** إلى **الكاش**، والمقصود به أموال الشريكين فقط.
5. إصلاح حسابات الزبائن والأقساط.
6. إصلاح شراء السيارات عن طريق شركة أو ممول.
7. منع أخطاء الربح عند اختلاف العملات.
8. جعل دفتر الأستاذ `financial_ledger` مصدر الحقيقة للحسابات المحاسبية الأساسية.

---

## تعليمات عامة وإلزامية

قبل أي تعديل:

1. افحص هيكل المشروع كاملاً وافهم منطق العمل الحالي.
2. لا تغيّر التصميم أو الشكل إلا في الأماكن المطلوبة صراحة.
3. لا تحذف أي وظيفة حالية غير مذكورة في الخطة.
4. لا تعيد هيكلة المشروع بالكامل بلا داعي.
5. لا تكسر أي API مستخدم حالياً في الواجهة إلا إذا عدّلت كل الأماكن المرتبطة به.
6. لا تعتمد على الواجهة فقط لإثبات صحة الحسابات.
7. لا تعتمد على mock أو localStorage في `src/api/tauri.ts` لإثبات صحة المحاسبة.
8. كل تعديل محاسبي يجب أن يكون قابلاً للتتبع في قاعدة البيانات.
9. بعد كل مرحلة، شغّل build واختبر السيناريوهات المحاسبية المذكورة.
10. لا تعتبر المهمة منجزة إلا بعد نجاح الاختبارات العملية والاستعلامات SQL الموجودة في نهاية الملف.

# القواعد التجارية النهائية

## 1. حذف خارج القاصة نهائياً

### القاعدة

لا يوجد في المشروع شيء اسمه:

```text
خارج القاصة
```

كل الأموال النقدية التي تدخل أو تخرج من النظام يجب أن تكون على حساب واحد فقط:

```text
قاصه
```

### المطلوب

احذف خيار **خارج القاصة** من:

- TypeScript types
- نماذج شراء السيارات
- نماذج بيع السيارات
- سجل القاصة
- الفلاتر
- أي Select أو Dropdown
- أي mock data
- أي شرط في Rust يعتمد على `خارج القاصة`
- أي migration أو بيانات قديمة

### المطلوب في قاعدة البيانات

أضف migration آمن عند تشغيل التطبيق يحول أي قيمة قديمة إلى `قاصه`.

نفّذ تحديثات مشابهة لما يلي، مع تعديل أسماء الجداول/الأعمدة حسب الموجود فعلياً:

```sql
UPDATE cars
SET purchase_payment_type = 'قاصه'
WHERE purchase_payment_type IS NULL
   OR purchase_payment_type = ''
   OR purchase_payment_type = 'خارج القاصة';

UPDATE cars
SET sale_payment_type = 'قاصه'
WHERE sale_payment_type IS NULL
   OR sale_payment_type = ''
   OR sale_payment_type = 'خارج القاصة';

UPDATE partner_transactions
SET payment_type = 'قاصه'
WHERE payment_type IS NULL
   OR payment_type = ''
   OR payment_type = 'خارج القاصة';

UPDATE financial_ledger
SET account_id = 'قاصه'
WHERE account_type = 'cash'
  AND (
    account_id IS NULL
    OR account_id = ''
    OR account_id = 'خارج القاصة'
  );
```

إذا كان بعض الأعمدة غير موجودة، لا تفشل migration؛ استخدم فحص schema أو تعامل آمن مع الأخطاء.

### المطلوب في Rust

أي قيد `cash` يجب أن يكون:

```text
account_type = cash
account_id = قاصه
```

ولا يسمح بتمرير `payment_type` من الواجهة كحساب نقدي إلا إذا كان يساوي `قاصه`.

الأفضل إنشاء دالة مركزية:

```rust
fn normalize_cash_account(_payment_type: Option<&str>) -> String {
    "قاصه".to_string()
}
```

واستخدامها في كل مكان يتم فيه تسجيل قيد cash.

### المطلوب في TypeScript

إذا وجدت شيئاً مثل:

```ts
purchasePaymentType: "قاصه" | "خارج القاصة";
salePaymentType: "قاصه" | "خارج القاصة";
```

استبدله بـ:

```ts
purchasePaymentType: "قاصه";
salePaymentType: "قاصه";
```

أو احذف `salePaymentType` و `purchasePaymentType` إذا لم يعد لهما داعٍ، لكن بدون كسر الواجهة.

## 2. أي إيداع لحساب نوعه شركة أو ممول لا يدخل القاصة نهائياً

### القاعدة

أي عملية **إيداع** لحساب نوعه:

```text
شركة
ممول
```

لا تدخل القاصة أبداً.

هذه القاعدة تشمل:

- إيداع يدوي
- إيداع غير يدوي
- إيداع من داخل صفحة الحساب
- إيداع من زر في أي شاشة
- إيداع ناتج عن تعديل عملية
- إيداع ناتج عن ترحيل
- أي transaction type يبدأ بـ `ايداع`
- أي transaction type يبدأ بـ `إيداع`
- أي transaction type يعامل كزيادة رصيد لحساب شركة أو ممول

### القيد الممنوع

ممنوع تسجيل هذا القيد لأي إيداع شركة أو ممول:

```text
Dr cash / قاصه
Cr funder
```

وممنوع أيضاً:

```text
Dr cash / قاصه
Cr payable
```

### التفسير التجاري

حسابات `ممول` و `شركة` ليست أموالاً تدخل القاصة عند الإيداع.

هي حسابات ذمم أو التزامات أو أرصدة متابعة.

لذلك لا يجوز أن تزيد القاصة بسبب أي إيداع في حساب ممول أو شركة.

### المطلوب في Rust

لا تجعل المنع موجوداً في مكان واحد فقط.

أنشئ دالة مركزية مشابهة:

```rust
fn is_deposit_type(tx_type: &str) -> bool {
    tx_type.starts_with("ايداع")
        || tx_type.starts_with("إيداع")
        || tx_type.starts_with("مقدمة")
        || tx_type.starts_with("استلام")
}

fn should_never_touch_cash_for_partner_deposit(kind: &str, tx_type: &str) -> bool {
    (kind == "ممول" || kind == "شركة") && is_deposit_type(tx_type)
}
```

ثم استخدمها قبل أي تسجيل قيد `cash`.

### السلوك الصحيح للإيداع

إذا كانت العملية مجرد إيداع أو تعديل رصيد لحساب ممول أو شركة:

الخيار الأول المقبول:

- تحفظ في `partner_transactions` فقط لأغراض كشف الحساب.
- لا تنشئ أي قيد في `financial_ledger`.

الخيار الثاني الأفضل محاسبياً:

تسجل كقيد غير نقدي، مثل:

#### ممول

```text
Dr opening_balance_adjustment
Cr funder / اسم الممول
```

#### شركة

```text
Dr opening_balance_adjustment
Cr payable / اسم الشركة
```

لكن لا تسجل أي `cash`.

### شراء سيارة عن طريق ممول أو شركة

شراء السيارة ليس إيداعاً.

إذا تم شراء سيارة عن طريق ممول:

```text
Dr inventory
Cr funder / اسم الممول
```

إذا تم شراء سيارة عن طريق شركة:

```text
Dr inventory
Cr payable / اسم الشركة
```

ولا يوجد أي قيد cash في الحالتين.

### السداد إلى ممول أو شركة

السداد ليس إيداعاً.

إذا تم دفع مبلغ إلى ممول أو شركة من القاصة:

```text
Dr funder/payable
Cr cash / قاصه
```

إذا تم السداد بتسوية خارجية أو عن طريق ممول آخر:

```text
لا تسجل cash
```

## 3. تغيير رأس المال إلى الكاش

### القاعدة

العميل لا يريد بطاقة اسمها **رأس المال** بمعنى محاسبي شامل.

المطلوب بطاقة اسمها:

```text
الكاش
```

ومعناها:

```text
أموال الشريكين فقط
```

أي أن الكاش لا يعني:

```text
الأصول - المطلوبين
```

ولا يعني:

```text
قيمة الشركة
```

ولا يعني:

```text
المخزون + القاصة + الذمم - الالتزامات
```

بل يعني فقط:

```text
مجموع أرصدة الشريكين من نوع شريك
```

### المطلوب في Backend

إذا كانت الحقول الحالية اسمها:

```rust
net_capital_iqd
net_capital_usd
```

فإما:

1. الإبقاء عليها مؤقتاً لتجنب كسر الواجهة، لكن تغيير معناها وتعليقاتها إلى `partner_cash`.
2. أو إضافة حقول جديدة أوضح:

```rust
partner_cash_iqd
partner_cash_usd
```

### معادلة الكاش

```sql
SELECT
  COALESCE(SUM(iqd_balance), 0) AS partner_cash_iqd,
  COALESCE(SUM(usd_balance), 0) AS partner_cash_usd
FROM partners
WHERE kind = 'شريك';
```

### المطلوب في Frontend

غيّر أي عنوان أو نص يظهر للمستخدم من:

```text
رأس المال
صافي رأس المال
```

إلى:

```text
الكاش
```

ويمكن إضافة وصف صغير:

```text
يمثل أموال الشريكين فقط
```

### ملاحظة مهمة

لا تستخدم معادلة الأصول والمطلوبين لهذه البطاقة.

إذا كان النظام يحتاج لاحقاً إلى بطاقة محاسبية شاملة، يمكن إضافتها باسم مختلف مثل:

```text
صافي أصول الشركة
```

لكن لا تستخدمها الآن بدلاً من الكاش.

## 4. إصلاح الزبائن والأقساط

### المشكلة المطلوب إصلاحها

يجب أن تكون حسابات الزبائن والأقساط دقيقة.

أي تسديد قسط يجب أن:

1. يزيد القاصة.
2. ينقص ذمة الزبون.
3. يظهر في كشف حساب الزبون.
4. يظهر في دفتر الأستاذ عند الحاجة.

### القاعدة المحاسبية

#### بيع سيارة كاش

```text
Dr cash / قاصه
Cr revenue
Dr COGS
Cr inventory
```

#### بيع سيارة بالتقسيط

عند البيع:

```text
Dr cash / قاصه              بالمقدمة
Dr receivable / اسم الزبون   بالمتبقي
Cr revenue                  بسعر البيع الكامل
Dr COGS                     بتكلفة السيارة + مصاريفها
Cr inventory                بتكلفة السيارة + مصاريفها
```

#### تسديد قسط

```text
Dr cash / قاصه
Cr receivable / اسم الزبون
```

#### إضافة باقي على زبون

إذا تم تسجيل مبلغ باقي على الزبون:

```text
Dr receivable / اسم الزبون
Cr adjustment أو revenue حسب طبيعة العملية
```

لا تسجل revenue إذا لم تكن العملية إيراداً حقيقياً.

### المطلوب في الكود

1. أضف فرع واضح في `record_partner_ledger_entries` لحسابات نوعها `زبون`.
2. لا تستبعد `تسديد قسط` من الدفتر إذا كان يمثل قبضاً فعلياً من الزبون.
3. عند تسديد قسط:
   - سجّل Dr cash.
   - سجّل Cr receivable.
4. أصلح `recalculate_partner_total` للزبون ليحسب الرصيد بشكل صحيح.

### معادلة رصيد الزبون

```text
رصيد الزبون =
البواقي
+ السحوبات
+ المبالغ المطلوبة منه
- التسديدات
- المقدمات
- المبالغ الواصلة منه
```

المطلوب أن يكون الرصيد النهائي واضحاً:

- موجب: الزبون مطلوب منه مبلغ.
- صفر: الحساب واصل.
- سالب: حسب سياسة المشروع، إما يظهر كرصيد زائد للزبون أو يمنع.

## 5. إصلاح شراء السيارة من شركة أو ممول

### القاعدة

`purchase_type = تمويل` يعني ممول.

```text
Dr inventory
Cr funder / اسم الممول
```

`purchase_type = شركة` يعني شركة.

```text
Dr inventory
Cr payable / اسم الشركة
```

### ممنوع

ممنوع أن تسجل عملية شراء من شركة تحت:

```text
funder
```

وممنوع أن تسجل عملية شراء من ممول تحت:

```text
payable
```

### المطلوب في Rust

ابحث في منطق شراء السيارة، خصوصاً `record_car_ledger_entries` أو أي دالة مشابهة.

يجب أن يكون المنطق واضحاً مثل:

```rust
if purchase_type == "تمويل" {
    // Cr funder
} else if purchase_type == "شركة" {
    // Cr payable
} else if purchase_type == "كاش" {
    // Cr cash / قاصه
}
```

## 6. إصلاح أرباح العملات

### المشكلة

ممنوع حساب الربح بهذه الطريقة إذا اختلفت العملة:

```text
profit = sale_price - purchase_cost
```

مثال خاطئ:

```text
سعر الشراء = 10000 USD
سعر البيع = 15000000 IQD
الربح = 15000000 - 10000
```

هذا خطأ محاسبي خطير.

### القاعدة

إذا كانت عملة الشراء مختلفة عن عملة البيع:

```text
لا توزع الربح تلقائياً
```

إلا بوجود سعر صرف واضح ومخزن مع العملية.

### الحل المطلوب الآن

في المرحلة الحالية:

1. إذا `purchase_currency != sale_currency`:
   - امنع توزيع الربح التلقائي.
   - أظهر رسالة واضحة للمستخدم.
   - لا تقسم الربح بين الشركاء.
   - سجّل العملية باعتبارها تحتاج مراجعة سعر صرف.

### الرسالة المقترحة

```text
لا يمكن احتساب أو توزيع ربح السيارة تلقائياً لأن عملة الشراء تختلف عن عملة البيع. يرجى إدخال سعر صرف واضح أو مراجعة العملية محاسبياً.
```

### الحل المستقبلي الاختياري

يمكن لاحقاً إضافة حقل:

```text
exchange_rate
```

ثم يحسب الربح كالتالي:

```text
cost_in_sale_currency = purchase_cost * exchange_rate
profit = sale_price - cost_in_sale_currency
```

لكن لا تضف هذا الحقل الآن إلا إذا كان تنفيذه آمناً ولا يكسر النظام.

## 7. توحيد مصدر الحقيقة المحاسبي

### القاعدة

يجب أن يكون `financial_ledger` مصدر الحقيقة لهذه البطاقات والحسابات:

- القاصة
- المخزون
- ذمم الزبائن
- المطلوب للممولين
- المطلوب للشركات
- الإيرادات
- المصاريف
- تكلفة المبيعات
- ميزان المراجعة

### أما `partner_transactions`

يستخدم لـ:

- كشف حساب الشخص
- عرض الحركات التفصيلية
- تاريخ العمليات
- ملاحظات العمليات
- مصدر مساعد للواجهة

لكن لا يجوز أن تكون بعض البطاقات مبنية على `financial_ledger` وبعضها على `partner_transactions` بطريقة تسبب تضارباً.

### المطلوب

راجع كل دوال الملخصات مثل:

- `get_financial_summary`
- `get_partners_totals`
- `get_cash_register_entries`
- أي دالة Dashboard
- أي دالة تقارير

وتأكد أن الحسابات المحاسبية الأساسية لا تتعارض.

# مراحل التنفيذ

## المرحلة الأولى: تنظيف خارج القاصة

1. ابحث في كامل المشروع عن:
   - `خارج القاصة`
   - `outside`
   - `salePaymentType`
   - `purchasePaymentType`
2. احذف خيار `خارج القاصة` من الواجهة.
3. غيّر الأنواع في TypeScript.
4. أضف migration يحول القيم القديمة إلى `قاصه`.
5. اجعل أي قيد cash يستخدم `account_id = قاصه`.
6. اختبر أنه لا توجد أي حركة cash خارج القاصة.

---

## المرحلة الثانية: تثبيت قاعدة شركة/ممول

1. أنشئ دالة مركزية تمنع دخول إيداعات شركة/ممول إلى cash.
2. طبّقها على كل عمليات `partner_transactions`.
3. راجع أي دالة تضيف ledger entries.
4. تأكد أن شراء السيارة من ممول/شركة لا يسجل cash.
5. تأكد أن السداد فقط هو الذي يمكن أن ينقص cash إذا كان من القاصة.

---

## المرحلة الثالثة: تغيير رأس المال إلى الكاش

1. غيّر النصوص في الواجهة من `رأس المال` إلى `الكاش`.
2. احسب الكاش من الشركاء فقط.
3. لا تستخدم معادلة الأصول ناقص المطلوبين.
4. حدّث أسماء المتغيرات إن أمكن بدون كسر الواجهة.
5. أضف تعليق واضح في الكود يشرح أن الكاش = أموال الشريكين فقط.

---

## المرحلة الرابعة: إصلاح الزبائن والأقساط

1. أضف منطق واضح لحسابات `زبون`.
2. أصلح تسديد الأقساط.
3. اجعل تسديد القسط يزيد القاصة وينقص receivable.
4. أصلح رصيد الزبون في الحسابات والكشوفات.
5. اختبر بيع تقسيط كامل مع مقدمات وتسديدات متعددة.

---

## المرحلة الخامسة: إصلاح شركة/ممول في شراء السيارات

1. راجع `purchase_type`.
2. افصل منطق `شركة` عن `تمويل`.
3. شركة = `payable`.
4. ممول = `funder`.
5. كاش = `cash / قاصه`.

---

## المرحلة السادسة: إصلاح أرباح العملات

1. افحص كل مكان يحسب:
   - profit
   - net profit
   - car profit
   - partner profit distribution
2. إذا اختلفت عملة الشراء والبيع:
   - لا توزع الربح.
   - لا تسجل أرباح للشركاء.
   - أظهر رسالة تنبيه.
3. إذا كانت العملة واحدة:
   - احسب الربح بشكل طبيعي.
   - وزع الربح حسب نظام المشروع الحالي.

### الإدخال

- سيارة بسعر شراء 10,000
- العملة IQD أو USD
- نوع الشراء كاش

### المتوقع

- القاصة تنقص بقيمة السيارة.

- لا تظهر شركة.
- لا توجد خارج القاصة.

- ديون الممولين

# استعلامات SQL إلزامية بعد التنفيذ

## 1. التأكد من حذف خارج القاصة

يجب أن يرجع 0 rows:

```sql
SELECT *
FROM financial_ledger
WHERE account_type = 'cash'
  AND account_id <> 'قاصه';
```

---

## 2. التأكد من عدم وجود خارج القاصة في السيارات

```sql
SELECT *
FROM cars
WHERE purchase_payment_type = 'خارج القاصة'
   OR sale_payment_type = 'خارج القاصة';
```

إذا كان أحد الأعمدة غير موجود، عدّل الاستعلام حسب schema.

النتيجة المطلوبة:

```text
0 rows
```

---

## 3. التأكد من عدم وجود خارج القاصة في معاملات الأشخاص

```sql
SELECT *
FROM partner_transactions
WHERE payment_type = 'خارج القاصة';
```

النتيجة المطلوبة:

```text
0 rows
```

---

## 4. التأكد أن إيداعات الممول والشركة لا تدخل القاصة

```sql
SELECT *
FROM financial_ledger fl
JOIN partner_transactions pt
  ON fl.reference_id = CAST(pt.id AS TEXT)
WHERE fl.account_type = 'cash'
  AND pt.kind IN ('ممول', 'شركة')
  AND (
    pt.tx_type LIKE 'ايداع%'
    OR pt.tx_type LIKE 'إيداع%'
    OR pt.tx_type LIKE 'مقدمة%'
    OR pt.tx_type LIKE 'استلام%'
  );
```

النتيجة المطلوبة:

```text
0 rows
```

---

## 5. توازن دفتر الأستاذ لكل عملة

```sql
SELECT
  currency,
  ROUND(SUM(debit), 2) AS total_debit,
  ROUND(SUM(credit), 2) AS total_credit,
  ROUND(SUM(debit - credit), 2) AS difference
FROM financial_ledger
GROUP BY currency;
```

المطلوب:

```text
difference = 0
```

لكل عملة، أو توضيح أي فرق بدقة إن كان بسبب بيانات قديمة تحتاج migration.

```sql

   OR description LIKE '%قسط%';
```

```sql
  AND sale_currency <> '';
```

ثم تأكد أنه لا توجد لها توزيعات أرباح تلقائية إلا إذا كان هناك سعر صرف واضح.

# معايير القبول النهائية

تعتبر المهمة مكتملة فقط إذا تحقق التالي:

- لا يوجد خيار `خارج القاصة` في الواجهة.
- لا يوجد `خارج القاصة` في `financial_ledger`.
- لا يوجد `خارج القاصة` في `partner_transactions`.
- كل قيد cash يستخدم `account_id = قاصه`.
- أي إيداع لشركة لا يدخل القاصة.
- أي إيداع لممول لا يدخل القاصة.
- شراء سيارة من شركة يسجل في `payable`.
- شراء سيارة من ممول يسجل في `funder`.
- تسديد قسط زبون يزيد القاصة وينقص receivable.
- بطاقة `الكاش` تعرض أموال الشريكين فقط.
- لا يوجد حساب ربح بين عملتين مختلفتين بدون سعر صرف.
- دفتر الأستاذ متوازن لكل عملة.
- الواجهة تعمل بدون أخطاء TypeScript.
- `npm run build` ينجح.
- تطبيق Tauri يعمل بدون أخطاء runtime.
