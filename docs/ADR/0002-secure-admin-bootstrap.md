# ADR-0002 — استبدال admin/admin الافتراضية بكلمة مرور عشوائية لمرة واحدة

- **التاريخ**: 2026-07-15
- **الحالة**: مقبول (Accepted)
- **المُقرّر**: فريق التدقيق الجنائي — جولة 11-B
- **المعرّف في `docs/BUG_REGRESSIONS.md`**: SECURITY-1
- **المراجع**: التكليف التنفيذي §7.1, `Instructions.md` §19, `src-tauri/src/lib.rs` الأسطر 849–1095, 2040–2095

## السياق

قبل هذا القرار، كان `init_db` يُنشئ قاعدة بيانات جديدة بحساب `admin` وكلمة مرور `admin` افتراضيةً. هذا نمط شائع في تطبيقات سطح المكتب المحلية لتسهيل الإعداد الأولي: المنشئ (operator) يُشغّل التطبيق، يدخل بـ `admin/admin`، ثم يُغيّر كلمة المرور. المشكلة: كثير من المنشئين لا يُغيّرون كلمة المرور الافتراضية، فيبقى النظام مفتوحًا لأي شخص يصل إلى الجهاز.

### المخاطر المحددة

1. **تسلّل كامل**: أي شخص يصل إلى التطبيق لأول مرة يدخل فورًا بـ `admin/admin`. هذا ينطبق على:
   - موظف غير مُصرّح يستخدم جهاز المحاسب.
   - مهاجم يصل للجهاز ماديًا أو عن بُعد (RDP).
   - برنامج خبيث يستهدف تطبيقات محاسبية.
2. **لا إجبار على التغيير**: حتى لو وُجد علم `must_change_password`، لم يكن مُطبّقًا في كل أوامر الكتابة. المنشئ يمكنه تجاهل التحذير واستخدام النظام بـ `admin/admin` إلى الأبد.
3. **ثغرة في الحزمة**: أي حزمة إصدار تحتوي قاعدة بيانات اختبار بـ `admin/admin` تُسلّم النظام للمشتري.
4. **تباين الحالات**: قاعدة بيانات جديدة (`count=0`) تُنشأ بـ `admin/admin`. قاعدة موجودة بـ `must_change_password=0` و`password_hash=admin_hash` تُترك كما هي. لا يمكن التمييز بين "المنشئ اختار كلمة مرور حقيقية وحدثها" و"المنشئ ترك `admin/admin` وعطل العلم".

### القيود البيئية

- التطبيق محلي سطح مكتب، لا شبكة. لا يمكن إرسال كلمة المرور عبر بريد إلكتروني للمنشئ.
- لا يمكن طلب كلمة مرور أثناء التثبيت لأن التثبيت عبر Tauri bundler لا يدعم prompts تفاعلية بشكل موثوق عبر كل أنظمة التشغيل.
- لا يمكن تخزين كلمة المرور في ملف على القرص (`initial_admin_password.txt`) لأن ذلك يُسلّم النظام لأي قارئ للملف.
- يجب أن يكون النظام قابل للاستخدام دون اتصال بالإنترنت.

## القرار

نستبدل `admin/admin` الافتراضية بنموذج **bootstrap آمن لمرة واحدة**:

### 1. توليد كلمة مرور عشوائية 24 حرفًا

الدالة `generate_one_time_admin_password()` في `lib.rs` (السطر 858) تولّد 24 حرفًا من `OsRng` (مولّد عشوائي آمن كريبتوغرافيًا) بأبجدية لا لبس فيها:

```rust
const ALPHABET: &[u8] = b"ABCDEFGHJKLMNPQRSTUVWXYZ23456789";
```

الأبجدية تتجنب `0`, `O`, `1`, `I` لقابلية النسخ اليدوي. الطول 24 حرفًا = 24 × log2(32) = 120 بت عشوائية، كافية لمقاومة هجوم brute force.

### 2. hash بكذا Argon2 ثم تخزين

كلمة المرور تُخزَّن كـ Argon2 hash في `users.password_hash` مع `must_change_password = 1`:

```rust
let one_time_pw = generate_one_time_admin_password();
let hash = hash_password_for_init(&one_time_pw)?;
conn.execute(
    "INSERT INTO users (id, username, password_hash, display_name, profile_image, must_change_password)
     VALUES (?1, ?2, ?3, 'مدير النظام', NULL, 1)",
    params![PRIMARY_ADMIN_USER_ID, DEFAULT_ADMIN_USERNAME, hash],
)?;
```

### 3. طباعة كلمة المرور على stderr مرة واحدة فقط

الدالة `announce_one_time_admin_password(username, password)` (السطر 874) تطبع:

```text
========================================
FAJR AL-WADI — ONE-TIME ADMIN BOOTSTRAP
Username: admin
Password: ABCDEFGHJKLMNPQRSTUVWXYZ23456789
This password will be shown ONLY ONCE.
You will be forced to change it on first login.
========================================
```

**لا تُكتب إلى أي ملف على القرص**. المُشغّل أو الـ Tauri installer أو الـ terminal operator يجب أن يلتقطها من `stderr` عند أول تشغيل.

### 4. لا إعادة توليد عند إعادة التشغيل

عند إعادة التشغيل، `init_db` يفحص:

- **Case A (count=0)**: قاعدة جديدة → توليد كلمة مرور عشوائية + hash + طباعة. (هذا ما يحدث أول مرة فقط.)
- **Case B (count=1, must_change_password=1)**: قاعدة موجودة لم تُغيّر كلمة المرور بعد → **لا** إعادة توليد، **لا** طباعة. الـ hash الأصلي يبقى. المنشئ يجب أن يستخدم كلمة المرور التي التقطها في أول تشغيل.
- **Case C (count=1, must_change_password=0)**: قاعدة موجودة غيّرت كلمة المرور. افحص إن كان الـ hash ما زال `admin/admin` legacy (ممكن إن كان المنشئ اختار `admin` ككلمة مرور جديدة). إن نعم، اطبع `[SECURITY] WARNING` على `stderr` دون إعادة تعيين صامتة (لأن ذلك قد يُغلق مُشغّلًا اختار كلمة مرور حقيقية فعلًا).

### 5. حظر `initial_admin_password.txt`

أي وجود لـ `initial_admin_password.txt` في شجرة المشروع يُفشل البناء عبر `scripts/check_artifact_hygiene.py`. هذا يمنع نمط "اكتب كلمة المرور لملف ثم احذفه" الذي يُخاطر بنسيان الحذف.

## العواقب

### إيجابية

- **لا تسلّل افتراضي**: قاعدة بيانات جديدة لا يمكن الدخول إليها إلا بكلمة المرور العشوائية المطبوعة. مهاجم لا يملك `stderr` التشغيل الأول لا يدخل.
- **إجبار التغيير**: `must_change_password=1` يمنع أي كتابة قبل تغيير كلمة المرور. حتى لو التقط مهاجم كلمة المرور العشوائية، المنشئ يُجبَر على تغييرها أولًا، فيُلغي فعاليّتها.
- **مقاومة brute force**: 120 بت عشوائية تجعل brute force مستحيلًا عمليًا.
- **مقاومة التسرّب**: لا يوجد ملف على القرص يحوي كلمة المرور. `stderr` يُلتقط مرة واحدة ثم يُختفي.
- **توافق مع `Instructions.md` §19**: "Forbidden: storing passwords in plain text". Argon2 hash فقط.
- **كشف الـ legacy**: قاعدة قديمة بـ `admin/admin` تُكشف بـ `verify_password(LEGACY_INSECURE_ADMIN_PASSWORD, stored_hash)` وتُحذّر.

### سلبية

- **خطر فقدان كلمة المرور الأولى**: إن لم يلتقط المنشئ `stderr` عند أول تشغيل، **لا يمكن استرجاع كلمة المرور** (لأنها hash). الحل: حذف قاعدة البيانات وإعادة التهيئة. هذا قد يُفقد بيانات إن أُضيفت بيانات قبل ملاحظة الفقدان.
- **عبء تشغيلي**: المنشئ يجب أن يفهم نمط "التقاط stderr". هذا غير بديهي للمستخدمين غير التقنيين. يجب توثيقه بوضوح في دليل التثبيت.
- **صعوبة الأتمتة**: أي سكربت CI/CD يُشغّل التطبيق لأول مرة يحتاج التقاط `stderr` لاستخراج كلمة المرور. هذا يُعقّد اختبارات E2E الحقيقية.
- **لا كشف للأجهزة المتعددة**: إن نُسخ التطبيق إلى جهاز ثانٍ مع نفس قاعدة البيانات، نفس كلمة المرور العشوائية تعمل. هذا مقصود (قاعدة البيانات تُهاجر مع التطبيق) لكنه يحتاج وعي المنشئ.
- **`must_change_password` legacy path**: الأوامر التي لم تُحدّث لتمرير `session_token` تسقط على النمط `None` الذي يفحص فقط وجود جلسة فعّالة + `must_change_password=0`. هذا أضعف من النمط `Some(token)`. الحل طويل المدى: تحديث كل أوامر الكتابة لتمرير `session_token` صراحةً.

### ملاحظة للمنشئين (Operators)

**يجب التقاط كلمة المرور من `stderr` عند أول تشغيل**. الخطوات:

1. ثبّت التطبيق عبر الـ installer أو `cargo tauri dev`.
2. شغّل التطبيق من terminal لرؤية `stderr` (على macOS/Linux: `./fajr-alwadi` من terminal؛ على Windows: شغّل من PowerShell أو CMD لرؤية الإخراج).
3. ابحث عن الكتلة المحاطة بـ `========================================` المُعلّمة `FAJR AL-WADI — ONE-TIME ADMIN BOOTSTRAP`.
4. انسخ اسم المستخدم (`admin`) وكلمة المرور (24 حرفًا).
5. سجّل الدخول. سيُجبرك النظام على تغيير كلمة المرور فورًا.
6. اختر كلمة مرور قوية (12+ حرفًا، حروف + أرقام + رموز). احفظها في مدير كلمات مرور.
7. **احذف** أي سجل لـ `stderr` (terminal history, log files) يحتوي كلمة المرور العشوائية.

إن فاتت كلمة المرور الأولى:

1. أوقف التطبيق.
2. احذف `fjr_alwadi_data.db` (إن لم تُضِف بيانات مهمة).
3. أعد التشغيل. ستُولّد كلمة مرور عشوائية جديدة.
4. التقط `stderr` هذه المرة.

إن أضفت بيانات قبل ملاحظة الفقدان، **لا تُحذف القاعدة**. تواصل مع فريق الدعم لإعادة تعيين كلمة المرور عبر SQL manual (تحديث `users.password_hash` بـ Argon2 hash جديد + `must_change_password=1`).

## بدائل مُعتبرة ومرفوضة

### بديل 1: كلمة مرور ثابتة معقدة بدل `admin/admin`

مرفوض. كلمة مرور مثل `Fajr@2026!Secure` تُنشر في الكود ويعرفها كل من يقرأ المصدر. هذا أسوأ من `admin/admin` لأنه يعطي إحساسًا زائفًا بالأمان.

### بديل 2: كلمة مرور في ملف `initial_admin_password.txt`

مرفوض. أي قارئ للملف يدخل النظام. حتى لو قلنا "احذف الملف بعد أول تشغيل"، المنشئ قد ينسى. الحل الأكثر أمانًا: لا تُخزَّن كلمة المرور على القرص أبدًا.

### بديل 3: طلب كلمة مرور أثناء التثبيت

مرفوض. Tauri bundler لا يدعم prompts تفاعلية موثوقة عبر كل أنظمة التشغيل (macOS .dmg, Windows .msi, Linux .deb/.AppImage). أيضًا، التثبيت الصامت (silent install) للنشر المؤسسي يتطلب عدم وجود prompts.

### بديل 4: عدم إنشاء حساب admin افتراضيًا

مرفوض. التطبيق غير قابل للاستخدام دون حساب admin. المنشئ لن يعرف كيف يُنشئ حسابًا عبر SQL مباشر. هذا يُفقد المستخدمين قبل البدء.

### بديل 5: Bootstrap عبر متصفح ويب محلي

مرفوض. يتطلب تشغيل HTTP server محلي أثناء الإعداد. هذا يُعقّد البنية ويُضيف سطح هجوم (CSRF, SSRF). التطبيق محلي سطح مكتب، لا حاجة لخادم ويب.

### بديل 6: OAuth/SSO خارجي

مرفوض. يتطلب اتصال إنترنت وخدمة خارجية. التطبيق يجب أن يعمل دون اتصال.
